Where we left off¶

Last post, we explored using Pipelines and GridSearchCV for training our models. The quick refresher is that GridSearchCV combines Cross-Validation (CV), a mechanism to test how well models generalise to data they have not seen before, with Grid Search (aka "Parameter Sweep"), a way to check which hyperparameters) yield the best performance. Pipelines are (IMO) not nearly as popular as they should be, as these provide an execution plan that gets followed by the GridSearchCV object. This means that, not only can you prevent leakage when doing feature scaling, but you can even let the model decide if it should scale features at all!

Combining these approaches will give you confidence that (given the algorithm, data, hyperparameters and processes you tested) you have picked the optimal model. So what's the problem? Well one fairly large problem is that we don't even know if the algorithm we picked is any good in the first place! For instance, your data clearly exhibits strong periodicity, but your algorithm is not flexible enough to capture it.

Unfortunately the CV process on its own cannot tell you this; you pick a metric to optimize and it will pick the hyperparameters that did the best, but it cannot tell you if it was "good". Adding more hyperparameters to test or even including multiple algorithms within the Pipeline/GridSearchCV object can help reduce the risk, but it does increase the run time substantially AND you still don't know if the performance is "good enough".

A good, principled solution in practice is to establish a baseline of performance and analyze the "Bias-Variance Trade-off" with Learning Curves). A baseline can provide you with a reference as to how well your more advanced approaches are actually improving the problems, while the Learning Curves can help you assess which regime (e.g. High Bias, High Variance) your model is currently in. There are some specific things to look at depending on if you are performing Regression or Classification, so we'll start with a Regression example, before moving on to a Classification example (probably in another post).

Regression Example¶

We'll get started using a Regression example. In this case, we will generate some 2D data using numpy.

In [1]:
# Creating some data for the example.  In this case we'll simulate some periodic data.  
import numpy as np
import matplotlib.pyplot as plt
# Set the Random Seed!
np.random.seed(1234)

# Create a function to make test and training data 
def make_data(linspace_ranges = (0,2), num_samples = 150):
    X = (linspace_ranges[1] - linspace_ranges[0]) * np.random.random_sample(num_samples) + linspace_ranges[0]
    y = 10 - 1/(X.reshape(-1,) + 0.1)
    y = np.random.normal(y, 1)
    X = X.reshape(-1,1)
    return X, y

X_train, y_train = make_data((0, 1))
X_test, y_test = make_data ((0, 1), 30)

#View the data
plt.figure(figsize=(14, 10))
plt.scatter(X_train, y_train)
plt.scatter(X_test, y_test, color = 'r')

Establishing Baselines¶

It is a common trap in modeling to focus only on the outcome metrics and when the outcomes seem good, we stop. But what if we had additional context for performance via a baseline? Consider the following example: Model A is trying to classify people into two groups and has an accuracy of 95% while Model B is also trying to classify objects into several types and has an accuracy of 65%. Though the tasks are different, which model sounds better? Nearly everyone you ask would select Model A because 95% > 65%. But what if you found out that just guessing the most common group of people would give you an accuracy of 99%? Conversely, what if guessing the most common object type only gives you an accuracy of 12%? Would you still pick Model A?

So-called "Dummy Regressors"/"Dummy Classifiers, simple models and pre-existing results provide practitioners with valuable context for model performance.

In this case, we'll use a Dummy Regressor predicting the median value and basic Linear Regression model. Both models are quite generic and neither is particularly advanced, but I personally always start a project doing something similar. Not only does this provide me a good baseline to compare against, but comparing these two approaches can possibly inform me about some structure to the data I should consider when fitting better models.

We'll generate the models and evaluate them both on the training set and testing set. And before we run the models, knowing what you know of our basic plot, you should ask yourself "Which if any of these estimators may perform well? Why?"

In [2]:
from sklearn.dummy import DummyRegressor
from sklearn.linear_model import LinearRegression

# There are several possible regressor strategies; I just picked the Median for fun.
dummy_reg = DummyRegressor(strategy = 'median')
# Fit a generic Linear Regressor with no bells or whistles.
simple_reg = LinearRegression()

# Fit the dummy and simple regression models
dummy_reg.fit(X_train, y_train)
simple_reg.fit(X_train, y_train)

# Predict both Training and Test data for both models
dummy_reg_train_pred = dummy_reg.predict(X_train)
simple_reg_train_pred = simple_reg.predict(X_train)

dummy_reg_test_pred = dummy_reg.predict(X_test)
simple_reg_test_pred = simple_reg.predict(X_test)

We can now pick several metrics to score the results with, such as Explained Variance, RMSE, R^2, etc...,. For this case, we'll use Mean Absolute Error, Root Mean Square Error and R^2.

NOTE: There are many other ways of doing this (and I prefer some over this), but I'll stick with using sklearn's PolynomialFeatures class as a way to stay entirely within sklearn's universe.

from sklearn.preprocessing import PolynomialFeatures
from sklearn.pipeline import make_pipeline

def PolynomialRegression(degree = 2):
    return make_pipeline(PolynomialFeatures(degree),
                         LinearRegression())

deg2_reg  = PolynomialRegression(2)
deg4_reg = PolynomialRegression(4)

The results confirmed our hunches:

  1. The second-degree polynomial did better than the simple regression model but did not perform as well on the test data.
  2. The fourth-degree polynomial had the strongest performance and generalized well to the test data.
  3. The 10th degree polynomial also performed well, basically matching the 4th degree polynomial.

Based solely on this evidence, we would select the 4th degree polynomial as our function since it had the best performance.

NOTE: Even if the metrics with the 4th and 10th degree polynomial were swapped, we'd still likely take the 4th degree polynomial since the performance was comparable and you should also choose the simplest model when there are ties (the Principle of Parsimony/Occam's Razor).

In this clearly contrived case, our visual inspection of the data and using our basic metrics makes it very clear which model is optimal. But what if we weren't in a 2D case? How can we quantify whether or not our models have high bias or high variance? Can we learn if adding more data is beneficial? The answer is Yes! We will use Learning Curves!

Learning Curves)¶

As mentioned earlier, learning curves give the practitioner the ability to quickly assess the bias and variance of the model. In addition, we can learn if more data is needed to improve the model.

There are several versions of this, the most common plotting errors on the y-axis as samples increase on the x-axis for Training and Validation data (i.e. when using the preferred "Train, Validate, Split" approach). However, I really like scikit's implementation. In this case, the "Out-of-Sample" is handled via a Cross-Validation operator.

scikit has great code to simultaneously plot learning curves with run times as well. To be clear and give credit where credit is due, I lifted the core part of this function and put it below. I didn't stick with their code (again, it's excellent) for a few reasons:

  1. I wanted to be able to change the scoring function; their code leaves it to the default scorer of your estimator.
  2. In regression, sklearn likes to find the optimal solution by finding the minimum of a function (why you see "neg_" in front of most regression scorers) and I thought it would be more intuitive for now to see functions being minimized.
  3. This was a great chance to pass a custom scorer through.

I've stripped out all of the timing functionality and some of the other bells and whistles, but the core functionality is the above code. Later, if you want to continue playing around with learning curves, I would definitely recommend using their code in the link.

In [10]:
from sklearn.model_selection import learning_curve

# Making a custom scorer for a couple reasons.
def make_rmse(estimator, X, y):
    y_pred = estimator.predict(X)
    return  np.sqrt(np.mean((y_pred - y)**2))

# Creating a modified plotting function to better handle the custom solver
def lc_plots(estimator, X, y, title, ylims = None, train_size_list = np.linspace(0.1, 1.0, 10), cv_func = 10, scoring_method = 'custom', y_axis_label = None):
    # Handle the customer scorer, or some other function before defaulting to the defaults
    if scoring_method == 'custom':
        scoring_func = make_rmse
    elif scoring_method is not None:
        scoring_func = scoring_method
    else:
        scoring_func = None
        
    # Create Learning Curve values
    num_samples, train_vals, cv_vals = learning_curve(estimator, X, y, 
                                                      train_sizes = np.array([x for x in train_size_list]), 
                                                      cv = cv_func,
                                                      n_jobs = -1,
                                                      scoring = scoring_func)
    # Make values for plots
    train_scores_mean = np.mean(train_vals, axis = 1)
    train_scores_std = np.std(train_vals, axis = 1)
    cv_scores_mean = np.mean(cv_vals, axis = 1)
    cv_scores_std = np.std(cv_vals, axis = 1)
    fig, ax = plt.subplots(figsize = (10, 8))
    if ylims is not None:
        plt.ylim(ylims)
    ax.fill_between(num_samples, train_scores_mean - train_scores_std,
                         train_scores_mean + train_scores_std, alpha=0.1,
                         color="r")
    ax.fill_between(num_samples, cv_scores_mean - cv_scores_std,
                         cv_scores_mean + cv_scores_std, alpha=0.1,
                         color="g")
    ax.plot(num_samples, train_scores_mean, 'o-', color="r",
                 label="Training score")
    ax.plot(num_samples, cv_scores_mean, 'o-', color="g",
                 label="Cross-validation score")
    ax.set_title(title, fontsize = 20)
    ax.yaxis.set_label_text( y_axis_label )
    ax.xaxis.set_label_text( '# of Samples' )
    plt.legend(loc = 'upper left', fontsize = 14)

lc_plots(dummy_reg, X_train, y_train, 'Dummy Regression', ylims = (0,4),

This is an excellent example of one of major pathologies of a high bias learner. Specifically:

  • The CV score (green line) is flat as we add data. This implies the learner is not able to learn from increased data
  • From our earlier work with metrics, we know that some estimators had RMSE's of about 1, so the level both Training and CV have converged to is quite high.
  • The Training score (red line) degrades quickly (not necessarily a problem) but then becomes flat. It may be increasing slightly, but in general this is indicative of something that is also not evolving with newer data.

  • Lastly, note that while the two lines are converging, there is a gap between the two.

Prediction Intervals (Regression Specific)¶

One last technique that can definitely be used for model validation would be using prediction intervals. We won't be delving into this in this notebook since it is not necessarily a function you can apply to every ML algorithm. In sklearn, I'm only familiar with making these intervals with Gradient Boosting Regressors/Random Forest Regressors and manually doing Linear Regression prediction intervals.

If you are doing one of these methods, it is possible to build prediction intervals and calculating various metrics associated to these (e.g. the proportion of test points within their prediction intervals) as another means to see if the model is generalizing well.

NOTE: Tree ensembles don't generalize well to inputs it's never seen before!

If you happen to be using a probabilistic framework, you'll be able to do more with your estimates as you'll have a full distribution to query regarding your out-of-sample predictions, but this is something we will cover much later. :)
